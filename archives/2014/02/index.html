One-page issue tracker Iteration 1

This is the first design of the One-Page issue tracker. I will be posting all iterations so everybody can see the design progress. This is the first iteration, and there is still a lot to do. Without further ado...

Walkthrough

Let's walk through it using the annotated version below:

  1. Logo which could be this issue trackers logo, or a company / project logo. And a dropdown menu that let's you change projects.
  2. Currently three menu items: Search, Export and Profile.
    • Search will let you search for issues. I'm not sure about the position because the search menu item is too far from the actual tickets. Something to look at for iteration 2.
    • Export opens a modal dialog to choose the export options.
    • Profile options, which can be used in the future if this issue tracker will support more users. For now it can be a menu with settings as update password / change name, etc.
  3. Filter bar to let you choose which issues will be displayed. This bar might be too far from the actual tickets. Something to look at for iteration 2.
  4. 3 boxes that will give you an overview of the project status. The first box shows how many issues are due today. The designs of these boxes are not done yet, and will be done in the next iterations. Other two boxes show you an overview of issues in testing and issues that require feedback.
  5. Header bar for the list of issues:
    • Issue number. Needed if you want to share the issue number with a developer
    • Severity. Self-explanatory.
    • Category.
    • Summary.
    • Description. I'm thinking of combining the Summary and Description fields into one field. I believe that it will take to much time to think of a summary for every issue. Something to think about in the next iteration.
    • Status
    • Assignee. For iteration 2, think about the use-case that multiple persons can be assigned.
    • Target fix date. Currently shown as full-date.
  6. The actual issues. I'm thinking of the following features:
    • On hover, an edit icon will be displayed and the fields can be edited inline.
    • Severity is displayed as a progress bar with only 3 stages. Low, Normal, High.
    • Mouse-over on the attachment icon will show the attachments, and over the comment icons will show the comments.
  7. WIP. I included this in Iteration so I could see if a 'add new issue bar' would work under or above the issue list. I think it will not, and I will work on it in the next iteration. Also the bar is too cluttered and doesn't allow to quickly add a new issue. Something to look at in the next iteration.
    • Idea: If I drag a file or email to on the issue list, it should create a new issue. If I drop it on an issue, it should be added as an attachment.

The good

It's always good to have the first iteration out. It helps me to focus on the next iteration and make sure the next iteration full of improvements. Feedback is welcome

The bad

Lots of improvements are needed. As you can see above, some parts are not designed, and for some parts already identified painpoints. And that is without any user feedback.

Designing a One-page issue tracker

According to Wikipedia, there are a lot of issue / bug trackers. If you're like me and you're managing a project, you likely tried most of them. The most common ones: Bugzilla, Mantis, Trac, all have the same in common: They get the job done, but are not the most pleasant to use. In this series of posts, I'm going to design (and maybe prototype) an issue tracker with the following goals:

Overview

Goals

  • One page based

The issue tracker will be one page based. The main page is an overview page with the list of issues. The user can open, close, update, comment on issues without navigating to another page.

  • Attachments

An image describing an issue is much clearer for the developer than a description. Images should be visible inline and it should be easy to add images to issues. No popups asking to name or describe an image.

  • Excelent export functionality

It should be able to generate a report that is easy to read and understand for managers. It should include the fixing progress, progress of the indivudual developers, etc. Managers don't have time to read a spreadsheet with rows full of issue numbers. It should be a spreadsheet that is nicely designed and gives a clear overview of the overall status. Ideally issues and images should be included.

Key assumptions

  • There will be one user adding, updating and monitoring issues.
  • The managed team is < 20 persons.
  • This issue tracker will be used for a website project.

Why

I would like to have some public projects. By documenting every step I hope others can learn from it, or if you think you a better idea, please feel free to leave feedback. With the feedback I hope to become a better User Experience designer. I could design a mobile app, another CMS or anything else, and maybe later I will do that, but currently I'm using an issue tracker and I have many ideas to improve it. In this series I will show you my design process, hopefully generating useful feedback, and in the end I will valuate if my ideas can improve issue tracking systems, or that the systems currently out there are as good as they can be.

Let's get started

Project Plan

  • Sitemap of planned pages
  • Content gathering
  • Wireframe / Interaction Design
  • Visual design
  • Prototype

Sitemap

Easy, since this is an One-page issue tracker. The following pages will be included:

  • Overview page (This is the main page, all other steps in the project plan will focus on this page)
  • Issue Detail page
  • Register / Login (Currently out of scope)

Content gathering

The second step is also not that difficult. Normally this step would include competitor analysis, interviews with stakeholders and users, etc. For an issue tracker it is a lot easier. I've looked at other issue trackers, and I've asked myself what do I need in this issue tracker so it will make my job as a project manager easer. I always like to not down the content so I can refer to it later when I lay it out on a wireframe. I've came up with the following:

  • Logo / Title
  • Project Name
  • Username
  • Search
  • Export
  • Overview of issue status
  • Daily / Weekly tasks
  • Issues:
  • ID
  • Severity
  • Category
  • Summary
  • Description
  • Attachments
  • Comments
  • Status
  • Assignee
  • Target fix date
  • Add new issue
